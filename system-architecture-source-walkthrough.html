AgenticOps 系统架构解析（含主要源码讲解）

这份文档用于给团队做系统讲解，重点解释“系统怎么跑起来、核心链路怎么实现、关键代码在哪、后续怎么演进”。内容基于当前仓库源码梳理，可直接用于技术分享或评审会。

Frontend: Vue3 + TS + Vite + Ant Design Vue Backend: FastAPI + SQLAlchemy Async Task: Celery Worker/Beat + Redis Ops Execute: Salt API + JumpServer API

01) 架构总览

表现层（Web）

  • 路由在 frontend/src/router/index.ts，将业务分成 dashboard/ops/rag/settings。
  • 统一 API 请求在 frontend/src/api/index.ts，自动带 Bearer Token。
  • Vite 代理 /api -> http://localhost:8000（frontend/vite.config.ts）。

控制层（FastAPI）

  • 应用入口 backend/app/main.py，挂载 auth/system/agent/ops 路由。
  • Ops 总路由聚合在 backend/app/api/ops/__init__.py。
  • 接口风格以 /api/ops/* 为主，前后端契约一致。

执行层（异步任务）

  • Celery 配置在 backend/celery_app.py，队列拆分为 salt/scheduler/jumpserver。
  • 执行逻辑在 backend/app/tasks/salt_tasks.py。
  • 定时扫描在 backend/app/tasks/scheduler.py，每分钟检查一次。

03) 关键业务链路（系统主要实现方式）

A. 手动触发任务链路（最核心）

TaskList.vue
点击“测试/触发”
POST /ops/tasks/trigger
tasks.py
execute_salt_command.delay
投递 Celery
salt_tasks.py
逐台执行 cmd.run_all
写 ops_task_execution_log
running/success/failed
LogList.vue 查询
/ops/logs/execution
关键代码位置：
- backend/app/api/ops/tasks.py -> trigger_task()
- backend/app/tasks/salt_tasks.py -> execute_salt_command()
- backend/app/api/ops/logs.py -> list_execution_logs()

B. 定时任务链路

  • Beat 每 60 秒触发 check_and_execute_tasks（backend/celery_app.py）。
  • backend/app/tasks/scheduler.py 读取启用任务，判断 next_run_at 是否到期。
  • 到期任务走与手动触发同一执行路径，保持一致性。
重点：手动触发和定时触发最终复用同一 Salt 执行任务，减少双实现风险。

C. 脚本分发链路

  • 脚本先上传到后端存储目录（scripts.py）。
  • 分发时读取脚本文本，base64 编码后远程落盘并 chmod +x。
  • 分发结果按服务器返回，支持部分成功（HTTP 207）。
关键接口：POST /api/ops/scripts/{script_id}/distribute
关键实现：backend/app/api/ops/scripts.py:505+

04) 核心数据模型（backend/app/models/ops.py）

模型 关键字段 作用
Server hostname, salt_minion_id, environment, status 维护目标机器信息，执行时选择 Salt target 与环境。
Script name, content(文件路径), script_type, timeout 脚本元数据；真实内容在文件系统，支持版本化。
ScheduledTask server_ids, cron_expression, command, script_id, enabled 定义“在哪些服务器、按什么规则执行什么命令/脚本”。
TaskExecutionLog task_id, server_id, status, command, error, duration 执行审计主表，一次任务会按服务器维度产生多条记录。

05) 运行拓扑与启动顺序

进程拓扑

  • API：uvicorn app.main:app --reload --port 8000
  • Worker：celery -A celery_app worker --loglevel=info -Q salt,scheduler
  • Beat：celery -A celery_app beat --loglevel=info
  • Frontend：npm run dev（5173）

配置关键点

  • backend/config.yaml 必须存在，否则导入配置即失败强依赖
  • Salt 环境参数来自 saltstack 节点。
  • JumpServer 参数来自 jumpserver 节点。
  • Redis 同时承担 Celery broker/result backend。

06) 当前质量与风险评估

已具备能力

  • 前后端契约清晰，核心运维链路完整可用
  • 手动触发与定时触发复用执行路径一致性好
  • 脚本管理支持版本对比与回滚可审计

主要风险

  • 后端缺少系统化测试和 CI高
  • 实时日志仅有“结果查询”，缺少 host 文件 tail中
  • 执行命令/路径安全校验仍需持续加强中
  • DB 初始化脚本默认强破坏性高

07) 演进建议（贴近你当前需求）

  • P0：新增 /api/ops/logs/execution/{log_id}/tail，手工输入日志路径 + 2s 轮询，先解决“不同主机路径不统一”的现实问题。
  • P0：前端日志详情增加“实时查看”区域（开始/停止、最后刷新时间、自动滚动）。
  • P1：构建最小 CI（前端 build + 后端 API smoke test + 核心任务触发回归）。
  • P1：为任务、日志高频筛选字段补索引与慢查询监控。
  • P2：在并发上来后再升级 SSE/WebSocket，避免过早复杂化。

08) 给别人讲解时建议顺序（15~20 分钟）

  1. 先讲目标：这是一个“统一运维控制面 + 异步执行引擎”的平台。
  2. 再讲分层：前端控制面、FastAPI 编排层、Celery 调度层、Salt/JumpServer 执行接入层。
  3. 重点讲一条主链路：点击触发任务 -> Celery -> Salt -> 执行日志 -> 前端查看。
  4. 讲源码地图：按本文第 02 节逐个文件说明职责，别人能马上定位代码。
  5. 最后讲风险与路线：实时日志、测试 CI、安全校验是下一阶段重点。
